What is the Spec-First Protocol?

Shifting the cognitive load of specification writing to an automated, iterative pipeline before execution begins.

The Spec-First Protocol (SFP) operates upstream of coding (before SDD or otherwise). It acts as an adversarial validation layer, ensuring requirements are captured, audited, and locked before execution begins.

Why Specs Matter

Starting execution without a specification is the most expensive shortcut in software development.

Traditional Problems SFP Solutions
Invisible Scope Creep
Prompting code changes directly without a base specification causes silent requirement expansion and makes change tracking impossible. 		Structured Interview
The discover skill captures and boundaries requirements before writing code, forcing alignment on scope boundaries.
Compounding Rework
Starting code generation with undefined requirements forces messy refactoring, structural breakages, and endless stop-and-ask prompts. 		Adversarial Audit
The audit skill reviews specifications for gaps and contradictions before execution starts, catching issues early.
Drift From Intent
Instructing execution agents based on fragmented chat threads forces LLMs to guess, producing code that misses the project owner's real needs. 		Zero Placeholder Invariant
Guarantees that every section is either fully written or omitted. No stubs, TODOs, or empty headings survive.
AI Context Drift
Feeding entire raw chat histories to code builders wastes token budgets and triggers code hallucination or structural collapse. 		Decoupled Lifecycle
Locks a dense, structured markdown spec. Downstream executors get exactly what they need, minimizing token bloat.

Step 1

discover

Initiates the protocol by conducting an interactive, requirements-gathering interview with the project owner. It compiles raw responses into a formal spec skeleton.

Primary Output: .sfp/discovery_notes.md, YYYY-MM-DD__SPEC_DRAFT.md
Required Gate: Explicit approval of initial scope from the project owner.

Skill Authoring Standards

Constraints and invariants enforced across the SFP ecosystem to guarantee token efficiency and processing reliability.

Constraint Name Limit / Threshold Technical Rationale
SKILL.md Length ≤ 500 lines (~5k tokens) Ensures execution agents can ingest core skill workflows inside compact reasoning loops without saturating context limits.
YAML Description ≤ 200 characters Strict boundary for reliable matching and dynamic skill routing engines (e.g. Claude's custom skill selector).
Reference Document Size ≤ 300 lines (~6 KB) per file Limits supplementary material to immediately absorbable lengths, preventing agent attention fragmentation.
Zero Placeholder Invariant 100% strict Prevents stale stubs, "TBD", or TODO items from surviving compilation gates, ensuring all code generators receive actionable instructions.
